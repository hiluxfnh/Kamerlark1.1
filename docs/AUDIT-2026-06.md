# KamerLark — Technical Audit & Ownership Report (June 2026)

## 1. Executive summary

KamerLark is a student-housing marketplace (listings, search, bookings, viewing
appointments, 1:1 chat, community/follows, support tickets, admin) built on
**Next.js 14 (App Router) + Firebase** (Auth, Firestore, Storage), fully
client-rendered.

At audit time the project **did not build** (broken import + an impossible
`output: "export"` configuration), had **secrets committed to git**, a
**privilege-escalation hole** in its security model, **Firestore rules that
could never have deployed** (invalid syntax) and that were **missing three
collections** the app writes to, and four UI components whose source files had
been **emptied to 0 bytes** while still being rendered (runtime crashes).
Three abandoned backends (Supabase/Express, MongoDB remnants, Firebase
hosting stubs) and four conflicting deployment targets coexisted in the repo.

All of the above are now fixed; `npm run build` passes cleanly for all 19
routes. The product is salvageable and roughly 70% feature-complete; the
biggest remaining risks are operational (rules deployment, secret rotation)
and structural (monolithic pages, zero tests, no payment/booking integrity).

## 3. Security audit

### Critical
1. **Secrets committed to git** — `.env`, `.env.production`, `server/.env`
   (MapTiler key, Supabase anon key; Firebase web config is public by design).
   *Fixed:* untracked + `.gitignore` hardened. **Action still required:**
   rotate the MapTiler key, restrict the Firebase API key (HTTP referrers) in
   Google Cloud Console, and purge history with `git filter-repo` before the
   repo is ever public.
2. **Admin privilege escalation** — admin status was read from
   `Users/{uid}.isAdmin`, a field **the user can write themselves** under the
   rules. *Fixed:* all admin checks now use the Firebase Auth custom claim
   (`token.claims.admin`); rules block any client write touching `isAdmin`;
   `scripts/set-admin-claim.js` provided to grant the claim.
3. **Firestore rules undeployable / wrong** — the committed rules used a
   nonexistent `has` operator (deploy would fail ⇒ whatever is live in the
   console is unknown and may be permissive test-mode). *Fixed:* valid `in`
   syntax; **deploy immediately**: `firebase deploy --only firestore`.

### High
4. **Bookings PII readable by any signed-in user** (name, email, phone,
   address). *Fixed:* only booker, listing owner, or admin can read.
5. **Missing rules for `supportTickets`, `appointments`, `Follows`** — under
   the committed default-deny these features were broken; under permissive
   live rules they were wide open. *Fixed:* explicit per-collection rules with
   participant scoping and schema validation on ticket creation.
6. **No Storage security rules in the repo** — profile pictures and chat
   images are governed by whatever was clicked in the console. **Action:**
   export/author `storage.rules` (scope writes to `profile_pictures/{uid}/**`
   and `images/chat/{uid}/**` by `request.auth.uid`) and add to firebase.json.

### Medium
- Hardcoded admin UID allowlist remains in `profile/page.js:350` (works only
  for that one account; not spoofable, but should migrate to the claim).
- No rate limiting / App Check — Firestore is callable by anyone with the
  public config. Enable **Firebase App Check** (reCAPTCHA v3/Enterprise).
- Booking/appointment documents trust all client-supplied values (price is
  not validated against the listing; no server-side invariant possible
  without a backend or Cloud Functions).
- `sessionStorage.setItem("user", true)` is vestigial pseudo-session state;
  harmless but misleading — Firebase Auth is the source of truth.

### Low
- Error messages from Firebase are surfaced verbatim to users (account
  enumeration hints). Map to friendly messages.
- `alert()` used for booking errors/success (UX + information leakage).

## 4. Architecture & stack assessment

**Verdict: keep Next.js + Firebase.** For a solo/small-team marketplace this
serverless BaaS stack is appropriate, cheap, and scales adequately. What was
wrong was not the stack but its configuration and the absence of any trusted
server-side layer.

- **Frontend**: Next.js 14 App Router, but every page is `"use client"` with
  Firestore fetching in `useEffect` — i.e., an SPA wearing Next.js clothes.
  Acceptable; do not fight it mid-stream. New code should prefer extracting
  data hooks (`useBookings`, `useChatRoom`) over more inline fetching.
- **UI**: MUI + Tailwind + styled-components + Emotion + CSS modules + two
  icon sets (Heroicons v1 *and* v2 + FontAwesome + MUI icons) — five styling
  systems. Converge on Tailwind + one icon set; remove the rest as pages get
  touched.
- **State**: none beyond local state; fine at this scale. Add a lightweight
  auth/user context to stop re-fetching `Users/{uid}` in every component.
- **Deployment**: was simultaneously targeting GitHub Pages (static), Vercel,
  Netlify, and Firebase Hosting. GitHub Pages workflow deleted (cannot host
  this app). **Pick one of Vercel or Netlify** — recommendation: Vercel
  (vercel.json present, README already documents it), then delete
  `netlify.toml`, `NETLIFY_DEPLOYMENT_GUIDE.md`, and the `netlify-cli`
  devDependency (~200 MB).
- **Quality gates**: builds run with `--no-lint` and
  `typescript.ignoreBuildErrors: true`. Re-enable incrementally.

## 5. Code quality

- **Monoliths**: `profile/page.js` (62 KB, ~1,750 lines), `roomdetails.js`
  (47 KB), `search/page.tsx` (39 KB), `listing/page.js` (35 KB). Split by tab
  /section into components + hooks as they are next touched.
- **Duplication**: the "update chatRoomMapping lastMessageTs" block is
  copy-pasted 4×; chat-room lookup logic duplicated across 5 files; admin
  gate duplicated 4× with three different mechanisms (now two).
- **Dead code removed**: `server/`, `src/app/services/`, `src/app/test/`,
  `"npm run build"/` folder, empty root scripts, GitHub Pages workflow,
  `bcryptjs`, `next-auth`.
- **No tests, no CI quality checks.** Highest-value first tests: Firestore
  rules tests with `@firebase/rules-unit-testing` (the rules are the entire
  security boundary of this product).
- Git history hygiene: commits like "er", "changes" ×6 — adopt conventional
  commits.

## 6. Roadmap

**P0 — operational security (this week)**
1. Deploy the corrected rules: `firebase deploy --only firestore` (rules + indexes).
2. Rotate MapTiler key; add HTTP-referrer restrictions to the Firebase API key.
3. Grant yourself the admin claim: `node scripts/set-admin-claim.js <sa.json> <your-uid>`.
4. Author and deploy `storage.rules`; enable App Check.
5. Choose Vercel; delete Netlify artifacts; set env vars in the dashboard.
6. Purge `.env*` from git history (`git filter-repo --path .env --path .env.production --path server/.env --invert-paths`) before making the repo public.

**P1 — product integrity (2–4 weeks)**
7. Booking integrity: availability check + status workflow (pending →
   confirmed/declined) enforced in rules; owner-side accept/decline UI is
   partially there — finish it.
8. Rules unit tests; wire `next lint` + `tsc --noEmit` into CI.
9. Replace `alert()` flows with the existing toast/snackbar components.
10. Real "backfill chat IDs" implementation or delete the placeholder page.

**P2 — sustainability (1–2 months)**
11. Decompose the four monolith pages into feature folders with hooks.
12. Converge styling on Tailwind; drop Emotion/styled-components/duplicate icon packs.
13. Auth/user context provider; remove redundant per-component user fetches.
14. Payments (Stripe) if bookings are to carry money; notifications (FCM).
15. Migrate remaining `.js` React files to `.tsx` incrementally; re-enable
    `ignoreBuildErrors: false`.

## 7. Q&A (Phase 8 mandate)

1. **Continue from current state?** Yes — after the P0 items. No rewrite needed.
2. **Refactor**: the four monolith pages, chat mapping updates, admin gating (done).
3. **Rewrite**: nothing wholesale; the `server/` experiment was deleted, not rewritten.
4. **Technologies appropriate?** Yes (Next.js + Firebase). The five-way styling stack is not.
5. **Architectural changes**: single deploy target; trusted layer for any
   money-touching logic (Cloud Functions or Next.js route handlers with
   firebase-admin); App Check.
6. **Essential security work**: deploy rules, rotate/restrict keys, storage
   rules, claim-based admin (done in code), history purge.
7. **If building today**: same stack, TypeScript-only, server components +
   route handlers with firebase-admin for writes that carry invariants
   (bookings), one styling system, rules tests from day one.
